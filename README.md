# Norde Maison — документация проекта (frontend)

### Что это за проект

Это клиентская часть (frontend) интернет‑магазина **Norde Maison** — SPA‑приложение, которое отвечает за интерфейс витрины, каталога, карточки товара, поиска, корзины, оформления заказа и личного кабинета.

Если backend — это “данные и бизнес‑правила”, то frontend — это **пользовательский сценарий**: как человек видит товар, выбирает вариант (цвет/размер), кладёт в корзину, оформляет заказ и уходит на оплату.

### Почему React и что это вообще такое

**React** — это библиотека для построения пользовательских интерфейсов: мы собираем приложение из компонентов (Header, карточка товара, форма checkout), а React эффективно обновляет экран при изменении данных.

Почему в дипломном проекте выбран React:

- **Компонентный подход**: удобно масштабировать витрину — переиспользовать карточки, модалки, формы, таблицы корзины.
- **SPA‑сценарий**: магазин — это много экранов и состояний (каталог → товар → корзина → checkout → профиль), React хорошо подходит под такие переходы без “перезагрузок страниц”.
- **Экосистема**: под типичные задачи e‑commerce есть проверенные решения (роутинг, работа с серверными данными, формы, анимации).

### Как frontend связан с backend

Frontend общается с backend по **REST API** через `axios`.

- **Базовый URL API**: сейчас зашит в `src/api/http.js`:
  - локально: `http://127.0.0.1:8000/api/`
  - (закомментировано) прод: `https://nordemaison.shop/api/`
- **Авторизация**: DRF Token. После логина токен кладётся в `локальное хранилище` и автоматически добавляется во все запросы как `Authorization: Token <token>`.
- **Формат данных**: фронт использует ответы API напрямую (React Query + `res.data`), ошибки показывает через нотификации.

Frontend Norde Maison — это “витрина магазина”: **каталог → поиск/фильтры → карточка товара → корзина → оформление → оплата → личный кабинет**.

Пользователь взаимодействует только с интерфейсом, а все данные (товары, варианты, остатки, корзина, заказы, статусы) подтягиваются из backend API.

Проект в этом репозитории — клиентская часть интернет‑магазина:

## 3) Основные библиотеки и зачем они нужны (по делу)

- **React 19**: база UI. Декларативные компоненты и предсказуемое обновление интерфейса при изменении данных (корзина, избранное, профиль).
- **Vite**: сборщик и dev‑сервер для быстрого разработки SPA (HMR), плюс удобная конфигурация алиасов и SCSS.
- **React Router DOM v6**: навигация по страницам и защита личных разделов (`ProtectedRoute`).
- **@tanstack/react-query**: работа с серверным состоянием “как с кэшем” — загрузка, повторные запросы, мутации и инвалидирование. В магазине это критично: корзина/избранное/заказы должны обновляться без ручной синхронизации.
- **axios**: HTTP‑клиент. Через interceptor автоматически приклеивается токен авторизации ко всем запросам.
- **react-hook-form**: формы с валидацией и аккуратной работой с полями (особенно checkout).
- **Sass (SCSS)**: стилизация; общие переменные и миксины подключаются глобально через Vite.
- **framer-motion**: плавные анимации UI (например, переключение вкладок профиля).
- **react-hot-toast**: быстрые нотификации об успехах/ошибках (добавили в корзину, ошибка оформления, и т.д.).
- **react-dadata**: UX для ввода адреса (подсказки/автодополнение).
- **lightgallery**: просмотр фотографий товара как галерея.
- **vite-plugin-svgr**: позволяет использовать SVG как React‑компоненты (иконки, UI).

## 4) Основные пользовательские сценарии

### Каталог → карточка товара

Смысл сценария: фронт рисует витрину на основе данных, которые отдаёт backend (категории/подкатегории/товары/варианты). Валюта прокидывается параметром, чтобы цены сразу приходили корректно.

2) В карточке товара пользователь выбирает **вариант** (цвет/размер) и количество.

### Избранное

Избранное сделано как “серверное состояние”: фронт запрашивает список и кэширует его через React Query. При добавлении/удалении вызывается мутация, затем кэш обновляется, и UI сразу меняется без ручных костылей.

### Корзина

Корзина привязана к пользователю, поэтому без авторизации работать не должна. На фронте это отражено так: если пользователь не вошёл — при попытке “Добавить в корзину” открывается модалка авторизации.

Дальше корзина живёт как кэшируемые серверные данные: изменение количества/удаление позиции — это мутации, после которых кэш корзины инвалидируется и подтягивается свежая версия.

### Оформление заказа → оплата

Оформление построено как “контролируемый сценарий”:

- перед показом формы фронт просит backend собрать **предпросмотр заказа** (состав корзины + данные по доставке/регионам);
- пользователь вводит данные и выбирает доставку;
- итог отображается сразу в UI, но финальная проверка и создание заказа происходят на backend.

Ключевой момент: **фронт не “рисует оплату” сам** — backend возвращает `payment_url`, и пользователь уходит на страницу оплаты (YooKassa).

Чтобы не плодить дубли, фронт отдельно проверяет “есть ли уже заказ, который ждёт оплаты”. Если есть — вместо создания нового показывается блок “ожидает оплаты” и кнопка “Оплатить”.

## 5) Роутинг (страницы приложения)

**Роутинг** — это “карта приложения”: по какому адресу в браузере какая страница показывается. В SPA на React это нужно, чтобы:

- пользователь мог переходить между разделами без перезагрузки страницы;
- можно было дать ссылку на конкретный экран (например, на товар `/product/123`);
- личные разделы можно было закрыть от гостей.

Маршруты описаны в `src/App.jsx`:

- `/` — главная
- `/about` — about
- `/men`, `/women` — витрина по полу
- `/catalog` — каталог
- `/product/:id` — карточка товара
- `/search` — результаты поиска

Защищённые маршруты (только для авторизованных):

- `/profile/*` — личный кабинет (заказы/избранное/профиль/выход)
- `/cart` — корзина
- `/checkout` — оформление заказа

Почему “такой подход с React” тут уместен: страницы — это просто компоненты, а роутинг решает, **какой компонент показать**. Дополнительно есть `ProtectedRoute`: если пользователь не вошёл, он не попадёт в корзину/оформление/профиль (перенаправление на главную).

## 6) Состояние приложения (как устроены данные)

В любом магазине есть “данные на экране”, которые постоянно меняются: кто вошёл, какая валюта, открыто ли окно поиска, что сейчас в корзине и т.д. Это и есть **состояние приложения** — “память”, от которой зависит интерфейс.

Чтобы не передавать одни и те же параметры через 20 компонентов подряд, в React используют **контекст**.

Если совсем просто:

- **Состояние** — это значения, которые могут меняться (например, `user` или `currency`).
- **Контекст** — это способ дать эти значения “всем нужным компонентам сразу”, без длинной цепочки пропсов.

В проекте есть 2 типа состояния:

- **Общее состояние интерфейса** (контексты): то, что нужно сразу в разных местах UI.
- **Данные с сервера** (корзина, избранное, заказы): это не “просто состояние”, это то, что приходит из backend и должно совпадать с ним.

- **Auth** (`src/context/AuthContext.jsx`):
  - хранит `user`, `isAuth`, управляет модалками авторизации/успеха/сброса пароля
  - при логине токен сохраняется в `localStorage`, при выходе токен удаляется
  - при старте приложения делается запрос “кто я” (`/auth/me/`), чтобы понять — пользователь вошёл или нет
- **Currency** (`src/context/CurrencyContext.jsx`):
  - валюта живёт в URL (`?currency=`) + `localStorage`
  - доступные: `rub`, `kzt`, `byn`
- **Search overlay** (`src/context/SearchContext.jsx`): открытие/закрытие поиска как часть интерфейса
- **Данные с сервера** (React Query): корзина, избранное, заказы
  - смысл React Query: хранить “последнюю актуальную версию” данных, уметь их обновлять и не путаться, когда пользователь что-то меняет (добавил товар, удалил товар, оформил заказ)

## 7) Как фронт и бэк реально “разговаривают”

Фронт — это “экран и кнопки”, бэк — “данные и правила”. Их связь выглядит так:

1) **Фронт просит данные** (товары, корзину, профиль) — бэк отвечает JSON.
2) **Фронт просит выполнить действие** (добавить в корзину, оформить заказ) — бэк проверяет правила (остатки, лимиты) и возвращает результат.
3) **Если нужна авторизация** — фронт отправляет токен, и бэк понимает, какой это пользователь.

### Как работает авторизация

- При входе бэк возвращает токен.
- Фронт сохраняет токен в `localStorage`.
- Дальше этот токен автоматически добавляется ко всем запросам.
- Поэтому корзина/оформление/профиль доступны только после входа.

### Как работает витрина (каталог/поиск/карточка товара)

- Фронт запрашивает список товаров и рисует карточки.
- При открытии товара запрашивает детальную информацию и показывает варианты (цвет/размер).
- Валюта выбирается на фронте и передаётся параметром в запросах, чтобы цены приходили в нужной валюте.

### Как работает корзина (и почему она “не ломается”)

- Добавление/удаление/изменение количества — это запросы на бэк.
- После каждого действия фронт заново берёт обновлённую корзину, чтобы то, что на экране, совпадало с сервером.

### Как работает оформление заказа и оплата

- Сначала бэк отдаёт предпросмотр заказа (что в корзине + варианты доставки).
- Фронт показывает форму, собирает данные и отправляет их на бэк.
- Бэк создаёт заказ и отдаёт `payment_url`.
- Фронт просто перенаправляет пользователя на страницу оплаты по этой ссылке.
- Если уже есть заказ “ждёт оплаты”, фронт показывает его и предлагает оплатить, а не создавать новый.

## 8) Структура проекта (важные папки)

Идея структуры простая: “где логика данных, где страницы, где кнопки”.

- `src/api/` — функции запросов к backend (авторизация, каталог, корзина, заказы). Тут живёт `axios` и базовые вызовы.
- `src/context/` — общее состояние интерфейса, которое нужно во многих местах (кто вошёл, валюта, открыт ли поиск).
- `src/hooks/` — “готовая логика” для страниц и компонентов: например, корзина или создание заказа как единый удобный хук.
- `src/pages/` — страницы приложения (то, что открывается по адресу в браузере).
- `src/components/` — переиспользуемые части интерфейса (хедер, футер, модалки, карточки товара).
- `src/assets/` — картинки и общие стили.

Vite alias: `@` указывает на `src` (см. `vite.config.js`).
